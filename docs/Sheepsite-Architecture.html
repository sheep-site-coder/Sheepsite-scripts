1. System Overview

SheepSite is a multi-building condominium website platform. It provides each building with:

  • Public and private Google Drive folder browsers
  • Resident login with per-building credential management
  • Resident database with reports (elevator, parking, resident, board) served from MySQL
  • Woolsy — an AI chatbot that answers resident questions from the building’s governing documents
  • Admin tools for building presidents/managers (user management, storage, Woolsy knowledge base)
  • A master admin interface for the SheepSite operator (Alain) to manage credits and platform-wide settings

All server-side logic lives in PHP scripts hosted at sheepsite.com/Scripts/. Building websites are served as subdomains of sheepsite.com (e.g., LyndhurstH.sheepsite.com) using a shared PHP template — building-site.php — with per-building configuration stored in config/{building}.json. Each subdomain’s index.php simply defines the BUILDING constant and requires the shared template. Google Drive access is handled by a Google Apps Script web app — PHP never calls Google APIs directly (see Section 4 for why). Resident data, reports, and email are handled by PHP + MySQL (see Section 5).

2. Actors & Roles

👑 SheepSite Operator (Alain)

Has master-level access to all buildings. Manages Woolsy credits, onboards new buildings, deploys code updates, and handles escalations that building admins cannot resolve.

Entry point: master-admin.php — authenticated with credentials/_master.json. Session key: master_admin_auth.

Can also: log into any building’s manage-users.php or admin.php using the master credential (shared bcrypt credentials file).

🏢 Building Admin (President / Property Manager)

Manages one building. Adds/removes resident accounts, imports owners from the Google Sheet, handles password resets, monitors Drive storage, and manages the Woolsy knowledge base.

Entry point: admin.php?building=X — authenticated with credentials/{building}_admin.json. Session key: manage_auth_{building}.

Sub-pages: manage-users.php, storage-report.php, woolsy-update.php, woolsy-admin.php

🏠 Resident (Owner)

Logs into the private area of their building’s site to browse private Drive files, view protected reports (elevator, parking, resident lists), chat with Woolsy, and change their password.

Entry point: display-private-dir.php?building=X — authenticated against credentials/{building}.json. Session key: private_auth_{building}.

👤 Public Visitor

No login required. Can browse the public Drive folder, view public reports (board of directors), and interact with the public Woolsy widget (deflections only — no AI cost).

Building Websites (subdomains of sheepsite.com)

Each building website is served as a subdomain (e.g., LyndhurstH.sheepsite.com). The subdomain’s document root contains a single index.php:

<?php
define('BUILDING', 'LyndhurstH');
require '/home/account/sheepsite.com/Scripts/building-site.php';

All pages, navigation, styling, and integrations are rendered by the shared building-site.php template. Per-building configuration (display name, header image, calendar URL, Facebook URL, property management contact) is stored in config/{building}.json and managed via building-detail.php.

Because the building site and all Scripts are on the same domain, PHP sessions are shared across pages — no cross-domain cookie issues. The Woolsy widget, private file browser, and resident portal all operate with the same session.

4. Google Apps Script Layer — Drive Only

Scope

Google Apps Script (GAS) is now used only for Google Drive access — file browsing, downloads, storage reports, and Woolsy document extraction. Resident data, reports (elevator, parking, resident, board), password reset emails, and owner import are all handled by PHP + MySQL (see Section 5). The building-script.gs per-building Sheets web app and its associated library functions are no longer called by PHP for DB-mode buildings.

The Problem: PHP Cannot Call Google Drive Directly

Google Drive API requires OAuth 2.0 authentication. On shared hosting (cPanel), there is no practical way to run a service account or manage OAuth refresh tokens securely — private key files on a shared server are a security risk, and the OAuth flow is complex to maintain.

The Solution: Google Apps Script as a Proxy

A Google Apps Script project deployed as a web app runs as Alain’s Google account. It has Drive access by virtue of being owned by the account that owns those files. PHP calls the web app via a simple HTTPS GET request — no OAuth, no credentials, no Google SDK.

5. MySQL Resident Database

Overview

All resident data is stored in a shared MySQL database on sheepsite.com. PHP scripts read and write the database directly — no Google Sheets or Apps Script involved. The Google Sheet is used only for the initial one-time import; after that, the database is the single source of truth.

DB-mode is active when both db/db.php and credentials/db.json are present on the server (which they always are on the live site). All buildings operate in DB-mode.

7. Authentication Architecture

Three Separate Auth Layers

LayerSession KeyCredential FileUsed By
Residentprivate_auth_{building}credentials/{building}.jsondisplay-private-dir, protected-report, chatbot-page, change-password
Building Adminmanage_auth_{building}credentials/{building}_admin.jsonadmin, manage-users, storage-report, woolsy-update, woolsy-admin
Master Adminmaster_admin_authcredentials/_master.jsonmaster-admin; also accepted as override in manage-users + admin

Credential File Format

Resident file (credentials/{building}.json) — array of user objects:

[
  {"user": "jsmith", "pass": "$2y$10$...", "mustChange": true},
  {"user": "mjones", "pass": "$2y$10$..."}
]

Admin file (*_admin.json) — array of admin objects (supports multiple admins):

[
  {"user": "admin",  "pass": "$2y$10$...", "email": "president@example.com", "mustChange": false},
  {"user": "jsmith", "pass": "$2y$10$...", "email": "jsmith@example.com"}
]

Master file (_master.json) — single object (unchanged):

{"user": "master", "pass": "$2y$10$..."}

All passwords are bcrypt-hashed. password_verify() is used for all checks. Helpers in db/admin-helpers.php: loadAdminCreds() (handles backward-compat single-object files), findAdminByPassword(), updateAdminEntry(), saveAdminCreds(). No plaintext passwords anywhere in the codebase.

Session Format

Building admin sessions (manage_auth_{building}) now store an array: {"user": "admin", "email": "..."}. Master admin assumed sessions store {"user": "_master", "email": ""}. Legacy sessions stored as plain true are invalidated on first load, forcing a re-login once.

Assume Admin Flow (Master Admin)

From master-admin.php, the Assume Admin button on each building card POSTs to master-admin.php with assume_building=X. This sets $_SESSION['manage_auth_X'] = ['user' => '_master', 'email' => ''] and redirects to admin.php?building=X. A blue “Viewing as Master Admin” banner appears at the top with a “Release ×” link that clears the session and returns to master-admin.php. No building admin password is required.

mustChange Flow

When mustChange: true is set on a resident account (e.g., after import or admin reset), the resident is redirected to change-password.php on every page load until they set a new password. They cannot reuse the temporary password. After changing, they are redirected to their original destination. The check runs on every authenticated page, not just at login, preventing bypass by direct URL access.

Building Validation

No hardcoded $buildings array is used for auth validation. A building is considered valid if its credential file exists on disk (credentials/{building}.json or credentials/{building}_admin.json). This means adding a building requires only creating the credential file — no code changes.

8. Woolsy Chatbot Stack

Architecture

Building Website (e.g., LyndhurstH.sheepsite.com)
  └─ chatbot-widget.js (floating Woolsy FAB)
       │
       ├─ GET chatbot-auth.php?action=whoami   ← checks session on every open
       │
       ├─ [not logged in] Public mode: deflections + inline login form
       │    └─ POST chatbot-auth.php?action=login  ← verifies credentials, sets session
       │
       └─ [logged in] Resident mode: inline chat panel
            └─ POST chatbot.php  ← question + history
                 ├─ faqs/{building}_rules.md (governing doc knowledge base)
                 ├─ faqs/{building}_docindex.txt (document/form directory)
                 └─ Anthropic API (Claude Haiku) → answer JSON

Because building sites are served from the same domain as Scripts (sheepsite.com), the private_auth_{building} session set by chatbot-auth.php is shared with display-private-dir.php, protected-report.php, and all other resident-facing pages — a single login covers the entire resident portal.

Public Widget (chatbot-widget.js)

Runs on the building website. Calls chatbot-auth.php?action=whoami (with cache: no-store) each time the FAB is clicked to get fresh session state. If not logged in, shows cheeky keyword-aware deflections and a “Log in to chat” button. Clicking that button opens a mini login form inside the floating panel — no page navigation, no popup window. The question that triggered the deflection is stored and automatically sent once login succeeds.

Resident Chat (chatbot-widget.js + chatbot.php)

When logged in, the widget renders a full chat panel. Questions are sent via fetch POST to chatbot.php, which:

  1. Checks credits (faqs/woolsy_credits.json) — blocks if exhausted
  2. Loads the building’s rules file and document index into the system prompt
  3. Calls Claude Haiku with the question and conversation history (last 6 exchanges)
  4. Calculates actual token cost and deducts from credits
  5. Returns the answer as JSON

Conversation history is maintained in JS (up to 12 messages). A logout button in the widget header calls chatbot-auth.php?action=logout and drops back to public deflector mode without any page reload.

10. Adding and Removing Associations

Adding a New Association

The automated path (recommended): in master-admin.php, click Add New Association in the System Tools grid. This opens building-detail.php?building=new, which:

  1. Phase 1: Enter building key, display name, state, community, and the three master IDs (template folder, association folder, template sheet). Click Create Drive Folders — GAS clones the template folder structure, sets Public sharing, copies the Owner DB sheet, and returns all folder/sheet IDs.
  2. Phase 2: A setup checklist appears with all values pre-filled: folder IDs, buildings.php snippet (copy button), admin URL, owner import walkthrough, and full footer script.

Three steps remain manual: set BUILDING_NAME in the sheet script, deploy it as a web app, and install the two time-driven triggers.

See NEW-SITE-GUIDE.html for the full walkthrough.

Removing an Association

In master-admin.php, click Remove Association in System Tools. The tool:

  • Shows a checklist of all server-side files that will be deleted (credentials, config, tags, Woolsy entries)
  • Deletes all MySQL rows for the building across the five tables (residents, car_db, unit_info, emergency, res_requests)
  • Requires typing the building key exactly to confirm — no undo
  • After deletion, shows a checklist of two remaining manual steps: remove the entry from buildings.php, and trash the Drive folders in the SheepSite Google account
Google Drive is not touched by this tool. Folder removal must be done manually.

11. Billing & Invoicing

Overview

SheepSite uses a file-based invoicing system. Invoices are JSON files stored in invoices/{building}/, one file per invoice. Three invoice types exist:

  • renewal — annual subscription invoice; generated manually or by cron 30 days before renewal
  • storage — storage tier upgrade; generated automatically when an upload is blocked by the limit
  • woolsy — Woolsy credit top-up; generated automatically when credits reach 90%

Each type stores the data needed to fulfil the service on payment (newBytes for storage, creditsToAdd for Woolsy). Payment can be made online via Stripe or by physical check marked paid manually by the operator.

Renewal Invoice Line Items

Built by buildLineItems() in invoice-helpers.php from the building config and config/pricing.json:

  1. Annual site fee (monthly price × 12)
  2. Discount — if discountPct is set, applied to the site fee only
  3. Domain renewal — if hasDomain is true and a domain price is set in pricing
  4. Storage upgrade — if the building’s storageLimit exceeds the default and matches a priced tier

Generating a Renewal Invoice

Manual: In building-detail.php, the Billing section shows a live preview of the next invoice and a Generate & Email Invoice button. Clicking generates the JSON, saves it, and emails it to the building’s contactEmail.

Automatic (cron): storage-cron.php checks each building’s renewalDate nightly. If renewal is within 30 days and no unpaid invoice exists for that period, it auto-generates and emails the invoice. unpaidInvoiceExists() prevents duplicates.

Threshold-Triggered Invoices (Storage & Woolsy)

These invoices are created automatically when a service limit is hit — no operator action needed:

  • Storage: when file-manager.php blocks an upload due to the storage limit, it calls checkStorageThreshold(building) in billing-helpers.php. If no email has been sent yet (storageLimitEmailSent flag absent), it: (1) picks the next storage tier from pricing, (2) calls createOpenInvoice() with invoiceType=storage and newBytes, (3) generates a 7-day billing token, (4) emails the contact a tokenized link to billing.php.
  • Woolsy: same flow triggered by checkWoolsyThreshold() in chatbot.php at 90% credit usage; creates an invoice with invoiceType=woolsy and creditsToAdd=10.

The billing token and invoiceId are stored in config/{building}.json under billingToken. The token expires after 7 days. Flags (storageLimitEmailSent, woolsyBillingEmailSent) prevent duplicate emails. Flags and token are cleared on payment.

Stripe Payment Flow

When a customer clicks the emailed link, they land on billing.php, which validates the billing token from the URL. For renewal invoices, customers reach billing-invoice.php via the paymentToken stored on the invoice. Both pages create a Stripe Checkout session via cURL and redirect the customer to Stripe. After payment, Stripe calls billing-webhook.php, which:

  1. Verifies the Stripe signature (HMAC-SHA256, 5-minute replay window)
  2. Routes by metadata.type: woolsy adds credits; storage sets storageLimit; invoice calls markInvoicePaid('stripe')
  3. Records the paymentIntentId in config/processed_payments.json to prevent double-fulfilment on retries

Marking an Invoice Paid (Check)

In building-detail.php, each unpaid invoice has a Mark Paid button. markInvoicePaid(building, invoiceId, paymentMethod) applies type-specific side effects:

invoiceTypeSide effects
renewalAdvances renewalDate one year from current date; clears suspended flag
storageSets storageLimit to invoice.newBytes; clears storageLimitEmailSent + billingToken
woolsyAdds invoice.creditsToAdd to woolsy_credits.json; clears woolsyBillingEmailSent + billingToken

In all cases: sets status: paid, paidDate, and paymentMethod (check/stripe) on the invoice JSON, then sends an HTML receipt email. The payment method is visible on invoice-view.php.

Invoice Email Design

HTML email with the SheepSite mascot logo, green brand header, line items table (light green header row), total block, and payment options footer. Receipt email has a green confirmation panel with invoice number, amount, payment date, and next renewal date. Both use invoiceEmailHtml() as a shared wrapper in invoice-helpers.php.

Discount

A per-building discountPct field in config/{building}.json (set in building-detail.php Configuration section) applies a percentage discount to the annual site fee only. It appears as a separate line item on the invoice. Domain and storage charges are never discounted.

Storage Cron Schedule

Recommended cPanel cron schedule: 0 3 * * * (3 AM daily). Command:

php /home/youraccount/sheepsite.com/Scripts/storage-cron.php

Can also be triggered manually via HTTP: https://sheepsite.com/Scripts/storage-cron.php?token=CRON_TOKEN. Results are logged to storage-cron.log in the Scripts folder.

12. Operator Procedures

This section documents recurring maintenance tasks performed by the SheepSite operator (Alain).

Updating the Woolsy Extraction Prompt

The Woolsy extraction prompt is the set of instructions sent to the Claude API when building or rebuilding a building's knowledge base (faqs/{building}_rules.md). As the system matures, the prompt may be improved to capture more topics, fix edge cases, or change the output format.

There are two prompts in woolsy-update.php:

  • Setup prompt — used when no rules.md exists yet (first run)
  • Update prompt — used when rules.md already exists and Drive files have changed, or a rebuild is forced

When to update the prompt

  • A topic category is missing or poorly handled (e.g. Woolsy can't answer questions about a known rule)
  • The output format should change (new sections, different heading style, more detail)
  • A bug in the extraction logic is found (e.g. section headings inconsistently named)

How to update the prompt

  1. Edit the prompt text in woolsy-update.php — the setup prompt and/or the update prompt, as needed
  2. Bump PROMPT_VERSION in both woolsy-update.php and admin.php (they must always match)
  3. Deploy both files to the server
What happens automatically: Every building's admin card reads the version stamp from its local rules.md and compares it to PROMPT_VERSION. Any building with an outdated stamp shows an amber warning banner: "Woolsy prompt updated (vN) — a rebuild is recommended". The building admin clicks "Rebuild now →" and the new prompt is applied. No manual tracking required — the version system flags every affected building automatically.

Version stamp mechanics

  • Every rules.md file begins with an HTML comment: <!-- woolsy_prompt_version: N -->
  • Files created before versioning was introduced have no stamp and are treated as v1
  • The stamp is written by woolsy-update.php automatically on every save
  • admin.php reads the stamp with getRulesVersion() — a local PHP function, no AJAX

Topping Up Woolsy Credits

  1. Log into master-admin.php (requires master credentials)
  2. Open the Woolsy Management card for the target building
  3. Adjust the credit balance and save

Credits are stored in faqs/woolsy_credits.json. 1 credit = $1 of Claude API cost (internal accounting). Buildings are billed externally at a markup. Hard stop at zero credits — Woolsy returns an unavailable message to residents until topped up.

Adding a New Topic to the Extraction Prompt

The prompt intentionally has no predefined topic list. Claude is instructed to extract every rule, policy, right, or obligation that an owner or resident would care about — the guiding test is "would a resident ask Woolsy about this?"

Two areas get explicit callouts in the prompt regardless:

  • Unit boundary definitions — what the owner owns vs. the corporation (doors, windows, balconies, HVAC, pipes). These are often buried in legal language but are among the most common resident questions.
  • Common element modification rules — approval process, dollar limits, what requires a board vote vs. owner vote. Also frequently buried in declarations or rules.

If you find that a topic is not being extracted reliably, the best fix is usually to add an explicit callout sentence to the prompt (e.g. "Always extract rules about X even if covered only briefly"), then bump PROMPT_VERSION and deploy. All buildings will be flagged for rebuild automatically.

Association Suspension and Renewal

Every resident-facing page includes suspension.php, which runs on each page load:

  1. Reads config/{building}.json
  2. If renewalDate is set and in the past, writes suspended: true to the config automatically
  3. If suspended is true, outputs a "Service Unavailable" page and exits — residents cannot access files, reports, search, or Woolsy

Admin pages (admin.php, manage-users.php, billing.php) are not blocked — the building admin can still log in and pay.

To reinstate a suspended association: log into master-admin.php, open the building’s Manage page (building-detail.php), and update the Renewal Date to a future date. Saving a future date automatically clears the suspended flag.

Master admin dashboard indicators:

  • ■ SUSPENDED — red badge + red card border when suspended: true
  • ⚠ RENEWAL DUE — amber badge when renewal date is within 30 days

Server Maintenance Notes

  • The credentials/ folder must remain writable by PHP (755 on the server)
  • faqs/ folder must also be writable (Woolsy update writes and overwrites files there)
  • After any PHP deploy, verify a known page loads before logging out of cPanel
  • The Apps Script GAS project does not need redeployment when PHP files change — only when dir-display-bridge.gs or building-script.gs changes
  • After any GAS master library change: Deploy → Manage deployments → New version (building scripts pick it up automatically if set to "latest version")
