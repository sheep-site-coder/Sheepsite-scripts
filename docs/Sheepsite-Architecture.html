1. System Overview

SheepSite is a multi-building condominium website platform. It provides each building with:

  • Public and private Google Drive folder browsers
  • Resident login with per-building credential management
  • Auto-generated reports (elevator, parking, resident, board) sourced from Google Sheets
  • Woolsy — an AI chatbot that answers resident questions from the building’s governing documents
  • Admin tools for building presidents/managers (user management, resident database, storage, Woolsy knowledge base, file management)
  • A master admin interface for the SheepSite operator (Alain) to manage all associations, Woolsy credits, storage, billing, and platform-wide settings
  • A billing system for Woolsy credit top-ups and storage upgrades via Stripe Checkout
  • License agreement (ToS) management with per-building click-through acceptance and signature audit trail

All server-side logic lives in PHP scripts hosted at sheepsite.com/Scripts/. Building websites are hosted separately (typically on Namecheap Website Builder) and integrate with SheepSite by pasting a footer script. Google Drive and Sheets access is handled by Google Apps Script web apps — PHP never calls Google APIs directly (see Section 4 for why).

2. Actors & Roles

👑 SheepSite Operator (Alain)

Has master-level access to all buildings. Manages Woolsy credits, onboards new buildings, deploys code updates, and handles escalations that building admins cannot resolve.

Entry point: master-admin.php — authenticated with credentials/_master.json. Session key: master_admin_auth.

Dashboard: One card per building showing Woolsy credit usage, storage usage, ToS status, and renewal date. System tools: Woolsy Overview, License Agreements, Pricing, Architecture Manual.

Per-building management: building-detail.php?building=X — Woolsy top-up, storage limit, config (siteURL, contactEmail, renewalDate, hasDomain), ToS status, billing history.

Can also: log into any building’s manage-users.php or admin.php using the master credential (shared bcrypt credentials file).

🏢 Building Admin (President / Property Manager)

Manages one building. Adds/removes resident accounts, imports owners from the Google Sheet, handles password resets, monitors Drive storage, and manages the Woolsy knowledge base.

Entry point: admin.php?building=X — authenticated with credentials/{building}_admin.json. Session key: manage_auth_{building}.

Sub-pages: manage-users.php, database-admin.php, storage-report.php, file-manager.php, woolsy-update.php, woolsy-manage.php

ToS gate: If the building is in ToS scope and the admin has not accepted the current version, admin.php redirects to tos-accept.php before showing the dashboard.

🏠 Resident (Owner)

Logs into the private area of their building’s site to browse private Drive files, view protected reports (elevator, parking, resident lists), chat with Woolsy, and change their password.

Entry point: display-private-dir.php?building=X — authenticated against credentials/{building}.json. Session key: private_auth_{building}.

👤 Public Visitor

No login required. Can browse the public Drive folder, view public reports (board of directors), and interact with the public Woolsy widget (deflections only — no AI cost).

4. Google Apps Script Layer — Why & How

The Problem: PHP Cannot Call Google Drive Directly

Google Drive API requires OAuth 2.0 authentication. On shared hosting (cPanel), there is no practical way to run a service account or manage OAuth refresh tokens securely — private key files on a shared server are a security risk, and the OAuth flow is complex to maintain.

The Solution: Google Apps Script as a Proxy

A Google Apps Script project deployed as a web app runs as Alain’s Google account. It has Drive and Sheets access by virtue of being owned by the account that owns those files. PHP calls the web app via a simple HTTPS GET request — no OAuth, no credentials, no Google SDK.

6. Authentication Architecture

Three Separate Auth Layers

Credential File Format

Resident file (credentials/{building}.json) — array of user objects:

[
  {"user": "jsmith", "pass": "$2y$10$...", "mustChange": true},
  {"user": "mjones", "pass": "$2y$10$..."}
]

Admin and master files (*_admin.json, _master.json) — single object:

{"user": "admin", "pass": "$2y$10$..."}

All passwords are bcrypt-hashed. password_verify() is used for all checks. No plaintext passwords anywhere in the codebase.

ToS Gate (Building Admin)

After the mustChange check passes, admin.php checks whether the building is in ToS scope (config/tos.json scope field: "all" or an array of building keys). If in scope and the accepted version in config/{building}.json is older than the current version, the admin is redirected to tos-accept.php. The gate repeats after every new ToS version is issued. Acceptance is recorded in both the building config and the append-only config/tos_signatures.json audit file.

mustChange Flow

When mustChange: true is set on a resident account (e.g., after import or admin reset), the resident is redirected to change-password.php on every page load until they set a new password. They cannot reuse the temporary password. After changing, they are redirected to their original destination. The check runs on every authenticated page, not just at login, preventing bypass by direct URL access.

Building Validation

No hardcoded $buildings array is used for auth validation. A building is considered valid if its credential file exists on disk (credentials/{building}.json or credentials/{building}_admin.json). This means adding a building requires only creating the credential file — no code changes.

7. Woolsy Chatbot Stack

Architecture

Building Website
  └─ chatbot-widget.js (public floating widget, deflections only)
       └─ window.open → chatbot-page.php?building=X&q=QUESTION (popup)
            └─ POST → chatbot.php (API endpoint)
                 ├─ faqs/{building}_rules.md (governing doc knowledge base)
                 ├─ faqs/{building}_docindex.txt (document/form directory)
                 └─ Anthropic API (Claude Haiku) → answer JSON

Public Widget (chatbot-widget.js)

Runs on the building website. Zero API cost — public questions get cheeky deflections with keyword-aware responses (pets, parking, noise, etc.) and a “Open Resident Chat” button. The question is preserved in the popup URL (?q=) and auto-sent after login.

Resident Chat (chatbot-page.php + chatbot.php)

chatbot-page.php handles the UI and login gate. chatbot.php is the API endpoint that:

  1. Checks credits (faqs/woolsy_credits.json) — blocks if exhausted
  2. Loads the building’s rules file and document index into the system prompt
  3. Calls Claude Haiku with the question and conversation history (last 6 exchanges)
  4. Calculates actual token cost and deducts from credits
  5. Returns the answer as JSON

Prompt Versioning

The extraction prompt (topic categories Claude looks for when building rules.md) is versioned via PROMPT_VERSION in woolsy-update.php and admin.php. When PROMPT_VERSION is bumped, the admin card detects the mismatch and shows a “Rebuild recommended” banner. This ensures all buildings benefit from improved topic coverage without manual tracking.

Credit Model

  • 1 credit = $1 of Claude API cost (internal accounting; Alain prices credits to buildings at a markup)
  • Default allocation: 1 credit per building
  • Hard stop at 100%: Woolsy returns “unavailable” message to residents
  • Low-credit warning at 80% usage shown in admin card
  • At 90%: chatbot.php calls checkWoolsyThreshold() from billing-helpers.php. If woolsyBillingEmailSent is not set in the building config, a billing email is sent to contactEmail with a tokenized link to billing.php. The flag is set to prevent repeat emails.
  • On payment: Stripe webhook adds purchased credits to woolsy_credits.json and clears the flag
  • Master admin can also top up manually via building-detail.php

9. Adding a Building

  1. In master-admin.php click + Add New Building → building-detail.php?building=new. Fill in the form to generate a buildings.php snippet. Copy and paste it into buildings.php manually, then deploy.
  2. Create the Google Sheet for the building, add building-script.gs bound to it, deploy as web app, install the two triggers
  3. Set up Drive folders (Public and Private) in the SheepSite Google account
  4. Visit forgot-password.php?building=X&role=admin&setup=1 to bootstrap the admin credential
  5. In building-detail.php, set siteURL, contactEmail, renewalDate, and hasDomain

  7. Log into admin.php?building=X, import residents into the database, sync user accounts, build the document index, run Woolsy setup
  8. If ToS scope includes the new building (or is set to "all"), the admin will be prompted to accept on first login

10. Operator Procedures

This section documents recurring maintenance tasks performed by the SheepSite operator (Alain).

Updating the Woolsy Extraction Prompt

The Woolsy extraction prompt is the set of instructions sent to the Claude API when building or rebuilding a building's knowledge base (faqs/{building}_rules.md). As the system matures, the prompt may be improved to capture more topics, fix edge cases, or change the output format.

There are two prompts in woolsy-update.php:

  • Setup prompt — used when no rules.md exists yet (first run)
  • Update prompt — used when rules.md already exists and Drive files have changed, or a rebuild is forced

When to update the prompt

  • A topic category is missing or poorly handled (e.g. Woolsy can't answer questions about a known rule)
  • The output format should change (new sections, different heading style, more detail)
  • A bug in the extraction logic is found (e.g. section headings inconsistently named)

How to update the prompt

  1. Edit the prompt text in woolsy-update.php — the setup prompt and/or the update prompt, as needed
  2. Bump PROMPT_VERSION in both woolsy-update.php and admin.php (they must always match)
  3. Deploy both files to the server
What happens automatically: Every building's admin card reads the version stamp from its local rules.md and compares it to PROMPT_VERSION. Any building with an outdated stamp shows an amber warning banner: "Woolsy prompt updated (vN) — a rebuild is recommended". The building admin clicks "Rebuild now →" and the new prompt is applied. No manual tracking required — the version system flags every affected building automatically.

Version stamp mechanics

  • Every rules.md file begins with an HTML comment: <!-- woolsy_prompt_version: N -->
  • Files created before versioning was introduced have no stamp and are treated as v1
  • The stamp is written by woolsy-update.php automatically on every save
  • admin.php reads the stamp with getRulesVersion() — a local PHP function, no AJAX

Topping Up Woolsy Credits

Automatic (via billing): When a building reaches 90% usage, a billing email is sent to the building’s contactEmail with a link to billing.php. The admin selects a credit quantity, pays via Stripe Checkout, and the webhook automatically adds the credits and clears the email-sent flag.

Manual (operator override):

  1. Log into master-admin.php
  2. Click Manage → for the building → Woolsy Credits section
  3. Enter credits to add and click Top Up

Credits are stored in faqs/woolsy_credits.json. 1 credit = $1 of Claude API cost. Buildings are billed at a markup configured in config/pricing.json.

Managing Storage Limits

Automatic (via billing): When a file upload is blocked by the storage limit, a billing email is sent to contactEmail. The admin selects a storage tier on billing.php, pays, and the webhook sets the new storageLimit in the building config immediately.

Manual (operator override): In building-detail.php → Storage section → set a new limit in MB. This also clears the storageLimitEmailSent flag so a new alert can fire if the limit is hit again.

Storage Cache Freshness

The storage limit check in file-manager.php uses a cached storageUsed value from config/{building}.json. This cache is updated whenever storage-report.php is loaded by an admin. A daily cron job (storage-cache-cron.php) refreshes the cache for all buildings automatically. If no cache exists, the upload proceeds unchecked (safe default for new buildings).

Managing the Terms of Service

  1. Edit the ToS document at docs/terms-of-service.html as needed
  2. Go to tos-admin.php → Issue New Version. This bumps the version number in config/tos.json and archives all current signatures
  3. On next admin login, all buildings in scope will be redirected to tos-accept.php to accept the new version
  4. To add or remove buildings from scope, use the enrollment checkboxes in tos-admin.php

Stripe Setup (first time)

  1. Create a Stripe account at stripe.com
  2. Get the live secret key (sk_live_...) from the Stripe Dashboard
  3. Add a webhook endpoint: URL = https://sheepsite.com/Scripts/billing-webhook.php, event = checkout.session.completed
  4. Copy the webhook signing secret (whsec_...) from the webhook settings
  5. Create config/stripe.json on the server: {"secretKey": "sk_live_...", "webhookSecret": "whsec_..."}
  6. Set actual prices in pricing-admin.php (storage tier $/month, credit price)

Adding a New Topic to the Extraction Prompt

The prompt intentionally has no predefined topic list. Claude is instructed to extract every rule, policy, right, or obligation that an owner or resident would care about — the guiding test is "would a resident ask Woolsy about this?"

Two areas get explicit callouts in the prompt regardless:

  • Unit boundary definitions — what the owner owns vs. the corporation (doors, windows, balconies, HVAC, pipes). These are often buried in legal language but are among the most common resident questions.
  • Common element modification rules — approval process, dollar limits, what requires a board vote vs. owner vote. Also frequently buried in declarations or rules.

If you find that a topic is not being extracted reliably, the best fix is usually to add an explicit callout sentence to the prompt (e.g. "Always extract rules about X even if covered only briefly"), then bump PROMPT_VERSION and deploy. All buildings will be flagged for rebuild automatically.

Server Maintenance Notes

  • The credentials/ folder must remain writable by PHP (755 on the server)
  • faqs/ folder must also be writable (Woolsy update writes and overwrites files there)
  • After any PHP deploy, verify a known page loads before logging out of cPanel
  • The Apps Script GAS project does not need redeployment when PHP files change — only when dir-display-bridge.gs or building-script.gs changes
  • After any GAS master library change: Deploy → Manage deployments → New version (building scripts pick it up automatically if set to "latest version")
